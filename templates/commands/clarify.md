# /clarify — 在已有 spec 时澄清模糊需求 (OPT-008)

> **触发条件**: spec.md 或 feature-spec.md 已存在但含模糊需求时使用
> **不要**与"无 spec 时探索"混淆 — 探索场景用户应直接对话, /clarify 用于"已有 spec 时澄清"
> **依赖**: 无前置命令，可在 /plan 之前使用

---

## 第一步：识别模糊点

读取 `docs/spec/` 目录下的 spec 文件，扫描 4 类模糊信号：

| 模糊类型 | 信号词 | 例子 |
|---------|--------|------|
| **未定义行为** | "可能", "大概", "或者", "类似" | "用户可能需要搜索" |
| **未指定边界** | "等", "之类", "若干" | "支持 CSV/JSON 等格式" |
| **未量化指标** | "快", "稳定", "友好" | "系统应该响应快" |
| **未明确优先级** | "也要", "顺便", "还要" | "顺便也加个暗黑模式" |

---

## 第二步：生成 Clarifications 段

在 spec.md 末尾追加以下结构（不要重写已有内容）：

```markdown
## Clarifications (auto-generated by /clarify at {timestamp})

### C-001: [模糊点 1 简述]
- **现状**: [从 spec 中提取的原文]
- **影响**: [不澄清会怎样]
- **候选方案**:
  - A) [方案 A]
  - B) [方案 B]
  - C) [方案 C]
- **建议**: [PESS 推荐, 含理由]
- **需用户确认**: [ ] / [x]

### C-002: ...
```

---

## 第三步：用户确认

输出以下问题模板等用户逐条回复：

```
已识别 N 个模糊点, 请逐条确认:
- C-001: 选 A / B / C? (默认: 推荐方案)
- C-002: 选 A / B / C? (默认: 推荐方案)
- ...
回复 'all default' 一键接受所有推荐方案。
```

---

## 第四步：更新 spec.md

用户确认后：
1. 把 Clarifications 段从"待确认"标记为"[x] 已确认"
2. 在每个 C-NNN 段下方追加 "**决定**: 选 X, 理由 ..." 一行
3. 不修改 spec.md 主体（避免污染原始设计）

---

## 第五步：日志到 activeContext

在 `memory-bank/activeContext.md` 追加：

```markdown
## [YYYY-MM-DD HH:MM] /clarify
- Spec: docs/spec/xxx.md
- 识别模糊点: N 个
- 用户确认: 全部 / 部分 / 默认
- 决定数: M 个
- 下一步: /plan
```

---

## 与"无 spec 时探索"的边界

| 维度 | /clarify | 手动 spec 探索 |
|------|----------|----------------|
| **触发** | spec.md 存在 | 无 spec |
| **目的** | 消除模糊 | 探索新领域 |
| **输出** | Clarifications 段 | spec 草稿 |
| **何时用** | /plan 之前 | /clarify 之前 |

> 注: PESS 当前没有独立 brainstorming 能力 (按辩论报告 §skills#5 拒绝),
> 无 spec 时用户应直接用 Claude 对话探索, 或先用 /specify (待实现)

---

## 失败处理

- **无 spec.md**: 输出 "❌ 未找到 docs/spec/*.md, 请先生成 spec (手动或 /specify 计划中)" 退出
- **spec 完全清晰** (0 模糊点): 输出 "✅ spec 无模糊点, 可直接 /plan" 退出
- **用户拒绝回答**: 标记 C-NNN 为 "pending" 状态, 提示继续 /plan 时会带歧义
